# 我把 DeepSeek 余额监控工具开源了：菜单栏常驻、低余额提醒、历史记录全都有

如果你经常用 DeepSeek API，你大概率经历过这种时刻：

- 写代码写到一半，模型突然报余额不足
- 想控制成本，但根本不知道今天已经消耗了多少
- 每次都要打开网页、登录、点进去看余额，流程又长又碎
- 明明只是想“顺手瞄一眼余额”，却被迫打断工作流

我自己被这个问题反复折磨之后，干脆做了一个专门给 mac 用户用的菜单栏工具：

**DeepSeek Balance Monitor for Mac**

现在，它已经正式开源。

- GitHub 仓库：
  - [github653224/DeepSeekBalanceMonitorForMac](https://github.com/github653224/DeepSeekBalanceMonitorForMac)
- 最新版本下载：
  - [Releases](https://github.com/github653224/DeepSeekBalanceMonitorForMac/releases)

这不是一个“为了开源而开源”的 Demo 项目。

它是一个我自己会真实长期使用的工具，所以它从第一天开始就不是只会显示一行余额数字，而是围绕**开发者真实使用场景**来设计的：

- 菜单栏常驻
- 一键查看余额
- 今日消耗展示
- 低余额提醒
- 日消耗提醒
- 历史记录
- 本地安全存储 API Key
- 原生设置窗口
- 可直接打包成 `.app` 和 `.dmg`

## 一、它解决的不是“查余额”，而是“减少打断”

大多数工具的问题不是功能不够，而是使用链路太长。

你本来正在：

- 写代码
- 调 Prompt
- 跑 Agent
- 做 API 调试

结果只是想确认一下余额，还要：

1. 打开浏览器
2. 找到网页
3. 登录账号
4. 进入控制台
5. 找余额位置

这个过程本身就已经把你的注意力切走了。

所以这个项目的核心设计目标，不是“把网页功能搬到桌面”，而是把最关键的信息压缩到**最短交互路径**里。

打开电脑，你在菜单栏里就能直接看到：

- 总余额
- 充值金额
- 赠送余额
- 今日消耗
- 上次查询时间
- 服务状态

这意味着它更像一个开发者的“余额雷达”，而不是一个传统意义上的桌面客户端。

## 二、界面我尽量做得克制，但要足够实用

### 1. 菜单栏主界面

![菜单栏主界面](images/screenshots/menu-bar.png)

这一屏不是为了炫 UI，而是为了让你在 3 秒内完成判断：

- 余额够不够
- 服务是不是正常
- 要不要立刻充值
- 要不要打开历史记录

这种设计背后的原则很简单：

**开发工具首先要快，其次才是花。**

### 2. 设置窗口

![设置窗口](images/screenshots/settings-window.png)

设置页支持：

- API Key 配置
- 查询间隔
- 余额预警线
- 日消耗提醒线
- 中英文切换
- CNY / USD 切换
- 开机自动启动

也就是说，这不是一个写死逻辑的小脚本，而是一个已经具备**可持续使用能力**的桌面工具。

### 3. 系统提醒

![低余额提醒](images/screenshots/low-balance-alert.png)

当余额低于你设定的阈值时，系统会直接弹出通知提醒。

这件事非常关键。

因为大多数人不是“不关心余额”，而是“忙起来根本想不起来看余额”。

通知机制的价值，就在于它把“被动查看”变成了“主动提醒”。

## 三、这次开源，我不是只放了代码，而是把完整工程链路也做出来了

很多开源项目的问题是：

- 仓库能看
- 代码能读
- 但你拉下来跑不起来
- 打不了包
- 发不了版本

我这次比较在意的是“工程完整度”。

所以除了功能代码，我还把下面这些东西一起整理好了：

### 1. Python 包结构整理

项目不是散落脚本，而是标准化包结构：

- `src/deepseek_balance_monitor_mac/`
- `core/`
- `infra/`
- `mac/`

好处是后面继续扩展时，不会越改越乱。

### 2. 原生 mac 菜单栏形态

这个项目不是 Electron 套壳，而是更贴合 mac 使用习惯的菜单栏工具思路。

这意味着：

- 常驻更自然
- 体积更克制
- 使用路径更短

### 3. 本地安全存储 API Key

我没有把 API Key 粗暴地明文塞进仓库或构建产物里，而是做了本地安全存储路径，确保：

- 开源仓库不会泄漏密钥
- 打包后的 `.app` 不会携带开发机私有 Key
- 用户第一次启动时走自己的本地配置流程

### 4. 支持 `.app`、`.dmg` 和 Python 包发布

这点对很多开源项目其实非常重要。

因为“能跑源码”和“普通用户能安装”完全是两回事。

现在这个仓库已经支持：

- 源码运行
- `.app` 打包
- `.dmg` 打包
- `wheel` / `sdist` 构建

### 5. GitHub Release 自动化已经跑通

这是我这次最满意的部分之一。

我没有停留在“代码能用”这一步，而是继续把发布链路打通了。

现在这个项目已经具备：

- 推送版本 tag
- 自动运行测试
- 自动构建 `.app`
- 自动构建 `.dmg`
- 自动构建 Python 包
- 自动生成校验文件
- 自动创建 GitHub Release
- 自动上传附件

这意味着后续发布版本时，维护成本会低很多。

## 六、为什么我选择把它开源

原因其实很简单。

我越来越不喜欢“工具只掌握在自己手里”的感觉。

一个真正有价值的小工具，如果它已经能解决真实问题，那它就值得：

- 被更多人下载
- 被更多人审查
- 被更多人二次开发
- 被更多人改造成自己的版本

而且这种工具型开源项目有一个很大的优点：

**用户反馈通常会非常直接。**

因为大家不是来围观概念，而是真的会拿去用。

只要有人开始真实使用，它就会快速告诉你：

- 什么地方体验不顺
- 什么功能最有价值
- 什么需求值得继续做

这比闭门造车高效太多了。
